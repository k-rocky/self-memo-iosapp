# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**Inner Orbit** - 感性を掘り起こすAIジャーナル

### プロダクトコンセプト
- **感性の発見**: 日常の一言メモとAIの問いかけを通じて、ユーザー自身の感性・価値観・熱量のパターンを少しずつ発見できる
- **思考を飛躍させる問い**: 正論や慰めではなく、「思考が少し斜め上に飛ぶ問い」をAIが返すことで、ユーザーの想像力と感性を掘り起こす
- **自分だけのログを育てる**: 日常の小さな気づきを積み重ねることで、自分の感性ログが形成され、週次サマリで傾向を発見できる

### プロダクトビジョン
現代の知的労働者は、安定した生活を送る一方で「自分は何に心が動くのか」「何に熱量を持てるのか」が分からなくなっている。
Inner Orbitは、日常のふとした気づきや違和感を起点に、AIが少し飛躍した問いを返すことで、ユーザーの感性・価値観・創造性を掘り起こす。
記録が積み重なることで「自分の感性の地図」が少しずつ描かれ、自分の人生に自分の色を取り戻す体験を提供する。
既存の日記アプリや自己啓発アプリとは異なり、正論や習慣化を押し付けず、感性や妄想を広げる余白を大切にする。

### 目的
- 日常の一言メモからAIが感性を刺激する問いを生成し、自己洞察を促す
- ログの蓄積を通じて、ユーザー自身の感性パターン・価値観・熱量の傾向を可視化する
- 「日記を続けられない」「自分の感性が分からない」という課題を解決する
- 夜に開きたくなるような、静かで没入感のある体験を提供する

## 成功指標(KPI)

### MVP検証指標（Phase 0〜1）

#### プライマリーKPI（定量）
- 継続率: 1週間で3回以上メモしたユーザー割合 → **50%以上**（MVP公開1ヶ月後）
- 翌日復帰率: 初回利用後に翌日以降も戻ってくる割合 → **40%以上**
- AI問い回答率: AIが生成した問いに対してユーザーが回答する割合 → **70%以上**
- 平均回答文字数: 1回あたりの平均回答文字数 → **50文字以上**
- 週次サマリ閲覧率: サマリが生成された際に閲覧するユーザー割合 → **60%以上**

#### セカンダリーKPI（定性）
- 「自分の感性が少し見えた」と感じるか（ユーザーインタビュー）
- AIの問いが浅くないか（問いの質評価）
- 日記より書きやすいか（比較評価）
- また使いたいと思うか（NPS）

### 個人開発としての成功指標
- 自分が1週間使いたいと思える
- 自分のログが10件以上溜まる
- 週次サマリを読んで腹落ちする発見が1つ以上ある
- 家族や友人1〜3人に触ってもらえる

## 機能要件

### コア機能（Phase 0: プロトタイプ）

#### 1. 一言メモ入力

**ユーザーストーリー**:
自分の感性が鈍っていると感じる知的労働者として、日常のふとした気づきを記録するために、シンプルな一言メモ入力機能が欲しい

**受け入れ条件**:
- [ ] アプリ起動から3タップ以内でメモ入力画面に到達できる
- [ ] テキスト入力フィールドが即座にフォーカスされ、キーボードが自動的に表示される
- [ ] 入力文字数に制限なし（ただしUIは短文を推奨するデザイン）
- [ ] 送信ボタンまたはキーボードの送信キーでメモを確定できる
- [ ] メモが空の場合は送信できない

**優先度**: P0（必須）

---

#### 2. AI問い生成

**ユーザーストーリー**:
自分の感性を掘り起こしたいユーザーとして、入力したメモに対してAIが感性を刺激する問いを受け取るために、AI問い生成機能が欲しい

**受け入れ条件**:
- [ ] メモ送信後、5秒以内にAIの問いが表示される（LLM APIレイテンシを考慮したローディング表示あり）
- [ ] 問いは1つだけ表示される（複数の問いは混乱を招くため）
- [ ] 問いのトーン：正論・慰め・カウンセリング風ではなく、「思考が少し斜め上に飛ぶ」方向性
- [ ] 問いの方向性バリエーション：妄想を広げる / 感情の奥を掘る / 価値観を見つける / SF・科学・未来・人生観に接続する
- [ ] 問いが生成されるまでの間、意味のあるローディング表現（「あなたのメモを読んでいます...」等）を表示する
- [ ] API呼び出し失敗時はエラーメッセージを表示し、再試行できる

**優先度**: P0（必須）

**AIプロンプト方針**:
```
入力例: 「結婚式で友達の熱量に嫉妬した」
生成例: 「あなたが羨ましかったのは、彼らの才能ですか？ それとも、
自分たちの人生を自分たちの色で表現している感じですか？」

入力例: 「宇宙のことを考えるとワクワクする」
生成例: 「もしあなたが一つだけ人類の未来に貢献できるとしたら、
それは"知ること"と"作ること"のどちらに近いですか？」
```

---

#### 3. 回答入力

**ユーザーストーリー**:
AIの問いに向き合いたいユーザーとして、自分の思考を自由に記述するために、自由記述の回答入力機能が欲しい

**受け入れ条件**:
- [ ] 問いを読んだ後、自由記述フィールドに回答を入力できる
- [ ] 回答はスキップ可能（問いだけ読んで終わる選択肢を提供）
- [ ] 入力中にAIの問いが常に見える状態を維持する
- [ ] 回答確定でログ保存が完了する

**優先度**: P0（必須）

---

#### 4. ログ保存

**ユーザーストーリー**:
自分の感性ログを育てたいユーザーとして、過去の記録を振り返るために、日々のメモ・問い・回答を保存・閲覧できる機能が欲しい

**受け入れ条件**:
- [ ] 以下のデータを保存する：日付・時刻 / ユーザーの一言メモ / AIが返した問い / ユーザーの回答 / 任意タグ（P0では省略可）/ 熱量スコア（P1で追加）
- [ ] ログ一覧画面でこれまでの記録を時系列（新しい順）で閲覧できる
- [ ] 各ログの詳細（メモ・問い・回答の全文）を確認できる
- [ ] Phase 0ではローカルストレージに保存（Core Data or SQLite）
- [ ] アプリ再起動後もデータが保持されている

**優先度**: P0（必須）

---

### 重要機能（Phase 1: MVP）

#### 5. 週次サマリ生成

**ユーザーストーリー**:
自分の感性パターンを発見したいユーザーとして、蓄積したログから自分の傾向を把握するために、AIが定期的に傾向をまとめる週次サマリ機能が欲しい

**受け入れ条件**:
- [ ] ログが5件以上蓄積されたら週次サマリを生成可能
- [ ] サマリは週に1回自動生成、または手動でも生成可能
- [ ] サマリの内容：反応しやすいテーマ / 感情パターン / 最近の頻出キーワード / 傾向の提示（断定ではなく「〜の傾向が見えます」という表現）
- [ ] サマリは心理診断のように断定しない（「あなたはこうです」→「こういう傾向が見えます」）
- [ ] サマリ生成中はローディング表示を行う
- [ ] 過去のサマリも閲覧可能

**優先度**: P1（重要）

---

#### 6. タグ機能

**ユーザーストーリー**:
ログを整理したいユーザーとして、メモにテーマやカテゴリを付与するために、タグ機能が欲しい

**受け入れ条件**:
- [ ] ログ保存時に任意のタグを1つ以上設定できる
- [ ] タグでログを絞り込み表示できる
- [ ] よく使うタグはサジェストされる
- [ ] タグは自由入力（プリセットなし）

**優先度**: P1（重要）

---

#### 7. 熱量スコア

**ユーザーストーリー**:
自分の感情強度を記録したいユーザーとして、メモを書いた時の感情の強さを定量化するために、熱量スコア入力機能が欲しい

**受け入れ条件**:
- [ ] メモ保存時に1〜5段階の熱量スコアを設定できる
- [ ] スコアは任意（設定しなくても保存可能）
- [ ] ログ一覧と週次サマリにスコアが反映される

**優先度**: P1（重要）

---

### 将来的な機能（Phase 2: Post-MVP）

#### 8. 簡易アカウント・クラウド同期

複数デバイスでのログ共有と、端末紛失時のデータ保護のために、アカウント認証とクラウドバックアップを提供する。

**優先度**: P2（できれば）

---

#### 9. TestFlight配布・フィードバック収集

他のユーザーへの体験提供と改善のために、TestFlightによる小規模配布とフィードバック収集機能を提供する。

**優先度**: P2（できれば）

## 非機能要件

### パフォーマンス
- アプリ起動時間: コールドスタートで2秒以内
- AI問い生成: LLM APIレスポンス後の表示まで100ms以内（API側のレイテンシは除く）
- ログ一覧表示: 100件まで0.5秒以内でレンダリング
- ログ保存: 保存操作から完了まで200ms以内

### ユーザビリティ
- 初回起動から最初のメモ入力完了まで3分以内で完了できる
- 1回の利用（メモ入力〜回答保存）が1〜3分で完結するUI設計
- 夜間利用を想定したダークモード対応（またはダーク基調のデフォルトテーマ）
- 余白・静けさ・少しSF感のあるUIデザイン（「静かな宇宙船のログ」世界観）

### プライバシー
- ログはユーザー本人だけが閲覧できる（Phase 2でのクラウド保存時も適用）
- AI処理のために送信される内容をUI上で明示する
- アカウント削除時にすべてのデータを削除できる（Phase 2）
- Phase 0〜1ではローカル保存のみ（外部送信はLLM API呼び出しのみ）

### セキュリティ
- APIキーをアプリに直埋めしない（バックエンド経由でLLM APIを呼び出す）
- 通信はHTTPS必須
- 認証を導入する場合はFirebase Auth / Sign in with Appleを使用（Phase 2）
- 個人ログへのアクセス制御（Phase 2でのマルチユーザー対応時）

### コスト管理
- LLM API使用料：1ユーザーあたり月100件のメモを想定してコスト試算を行う
- 週次サマリは毎回ではなく週1回または手動実行にして不要なAPI呼び出しを削減
- MVP段階でのAPI使用量の上限設定を検討する

### 信頼性
- Phase 0（ローカル保存）: データ損失ゼロを目標とし、Core Dataの自動バックアップを活用
- Phase 2（クラウド保存）: 自動バックアップと冗長化を実装

## スコープ外

### MVP（Phase 0〜1）で明示的に除外する項目
- SNS機能・他ユーザーとの共有
- 高度なグラフ・データ可視化
- ゲーミフィケーション（バッジ、ストリーク等）
- 課金・サブスクリプション
- 複雑なプロフィール診断
- 長期的なパーソナライズAI（ユーザー専用モデルのファインチューニング等）
- Apple Watch連携
- 音声入力
- カレンダー連携
- プッシュ通知（リマインダー）
- ウィジェット

**理由**: 最初に検証すべきは「一言メモ → AIの問い → 自己洞察」という体験が刺さるかどうかであり、周辺機能は体験の質が確認できてから追加する。

## 開発フェーズ計画

### Phase 0: プロトタイプ（個人検証）
**目的**: 自分が使って刺さる体験か確認する

**機能**:
- メモ入力
- AI問い生成
- 回答入力
- ローカル保存（Core Data）
- ログ一覧

**技術スタック**:
- iOS: SwiftUI
- ローカルDB: Core Data
- バックエンド: Node.js (Express) — 開発者のローカルマシンで起動し、iOSシミュレーターから接続
- LLM: Claude API（Anthropic）

**APIキー管理方針（Phase 0）**:
LLM APIキーはバックエンドの環境変数（`.env`）で管理し、iOSアプリのバイナリには含めない。
Phase 0では開発者のMac上でバックエンドを起動し、iOSシミュレーターから `http://localhost:3000` に接続する。
実機テスト時はローカルネットワーク経由で接続するか、無料枠のクラウドサービス（Railway等）を利用する。

### Phase 1: MVP（小規模ユーザー検証）
**目的**: 1週間使える状態にする

**追加機能**:
- 週次サマリ
- タグ
- 熱量スコア
- UI改善
- プロンプト改善

### Phase 2: 小規模配布（他者への検証）
**目的**: 他人にも刺さるか確認する

**追加機能**:
- 簡易アカウント認証
- クラウドバックアップ
- TestFlight配布
- フィードバック収集機能
